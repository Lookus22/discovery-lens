What is Discovery Lens? A Streamlit tool that helps product managers at any tech company (SaaS, mobile app, web product) synthesise discovery data — interviews, app reviews, support tickets, usability notes — into a structured Opportunity-Solution Tree grounded in evidence. Any PM sets their product goal, uploads their discovery files, and the tool clusters insights, maps opportunities in JTBD language, and shows exactly which evidence justifies each decision.

🎯
Product in one sentence: Discovery Lens is a generalist tool that helps any product manager in a tech company transform raw qualitative and quantitative discovery data into a structured, evidence-backed Opportunity-Solution Tree — making prioritisation transparent and decisions traceable.
Core user flow (MVP)
1
Set product goal
PM enters product name + goal statement (e.g. "Increase 30-day retention for new users"). This threads through all LLM prompts.
2
Upload discovery files
Multi-file upload: PDF, TXT, CSV, DOCX. Each file tagged with source type (interview / review / ticket / usability).
3
Automatic extraction & chunking
System extracts text from each file, splits into 2–4 sentence chunks, stores source metadata per chunk.
4
Embed + cluster
sentence-transformers encode chunks → K-means clusters thematically similar content. Optimal k selected automatically.
5
LLM frames opportunities
Groq/Llama receives each cluster + goal context → returns opportunity in JTBD format + 2–3 solution suggestions + risky assumptions.
6
OST renders
Interactive tree: Goal → Opportunities → Solutions → Assumptions. Nodes clickable. PM can edit, add, or remove any node.
7
Traceability on demand
Click any opportunity → see the top source quotes that generated it, labelled by file and source type.
8
Export & decide
PM views priority matrix + evidence heatmap, reviews assumption register, downloads OST as JSON.

Non-functional requirements
Performance
Full pipeline (upload → OST render) must complete in <30s for demo files up to 2MB.
Embedding + clustering must be cached (@st.cache_data) so re-runs are instant.
Groq free tier limit: ~30 req/min — batch cluster processing accordingly.
Constraints
Single-user, single-project session. No authentication.
Max ~500 text chunks per session (performance on Streamlit Cloud).
No database. State lives in st.session_state + JSON export.
All dependencies must run on Streamlit Community Cloud (no GPU).

Data contracts between pipeline modules
extractor.py
Input: uploaded file (Streamlit UploadedFile), source_type (str)
Output: raw_text (str)
chunker.py
Input: raw_text (str), filename (str), source_type (str)
Output: List of dicts: {chunk_id, text, filename, source_type}
embedder.py
Input: chunks (List[dict])
Output: embeddings (np.ndarray shape [n_chunks, 384])
clusterer.py
Input: chunks (List[dict]), embeddings (np.ndarray)
Output: clusters (List[dict]): {cluster_id, representative_chunks[3], all_chunk_ids}
llm.py
Input: clusters (List[dict]), goal (str)
Output: ost_json: {goal, opportunities[{jtbd, solutions[{label, assumptions[{text, risk}]}], priority_score}]}
Team working agreements

Each person answers 3 questions:
  • What did I complete since yesterday?
  • What will I work on today?
  • Any blocker that needs the PM's decision?

PM closes with: any priority changes, any scope decisions, any help needed across team members. Run it async in Slack if not meeting in person.
Definition of done (per task)
A task is done when:
  • Code is merged to main (via PR, not direct push)
  • It doesn't break any existing feature
  • PM or one other teammate has seen it work
  • The checkbox in this plan is ticked

Partial work = not done. If a task is too big, split it in the daily.
Git workflow
  • One branch per feature: feature/ost-tree, fix/embedding-crash
  • Never push directly to main
  • Small PRs (1 feature = 1 PR). Easier to review.
  • Tag the PM in any PR that changes the pipeline output format — it affects everyone
  • Use GitHub Issues for bugs. Label: P1 blocker / P2 important / P3 nice-to-have
PM scope decisions (do not build without PM sign-off)
The PM must approve before the team builds:
  • Any change to what the LLM outputs
  • Any new page or view added to the app
  • Any change to the OST data structure
  • Any new dataset or data source added

For anything else: build it, show it in the daily, PM reviews. Speed over process on small things.

Risk register
RiskLevelMitigation

LLM output inconsistency: Groq/Llama produces malformed or non-JTBD output Medium Fix JSON output schema in system prompt + few-shot examples in Week 1. Add try/except around JSON parsing. Use llama-3.3-70b as fallback if 8B fails.
OST tree visualisation complexity: streamlit-agraph or pyvis bugs/layout issues Medium Default to custom HTML/JS tree (simpler, more predictable). Only use a library if the custom component fails after 1 day of trying.
