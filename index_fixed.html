What is Discovery Lens? A Streamlit tool that helps product managers at any tech company (SaaS, mobile app, web product) synthesise discovery data — interviews, app reviews, support tickets, usability notes — into a structured Opportunity-Solution Tree grounded in evidence. Any PM sets their product goal, uploads their discovery files, and the tool clusters insights, maps opportunities in JTBD language, and shows exactly which evidence justifies each decision.

Product in one sentence: Discovery Lens is a generalist tool that helps any PM transform raw qualitative discovery data into a structured, evidence-backed Opportunity-Solution Tree — making prioritisation transparent and decisions traceable.
Core user flow (MVP)
1
Set product goal
PM enters product name + goal statement. Threads through all LLM prompts.
2
Upload discovery files
Multi-file upload: PDF, TXT, CSV, DOCX. Each file tagged with source type.
3
Automatic extraction & chunking
System extracts text, splits into 2–4 sentence chunks, stores source metadata.
4
Embed + cluster
sentence-transformers encode chunks → K-means clusters thematically similar content.
5
LLM frames opportunities
Groq/Llama returns opportunity in JTBD format + solutions + risky assumptions.
6
OST renders
Interactive tree: Goal → Opportunities → Solutions → Assumptions. PM can edit.
7
Traceability on demand
Click any opportunity → see the top source quotes that generated it.
8
Export & decide
PM views priority matrix + evidence heatmap, downloads OST as JSON.

Data contracts between pipeline modules
extractor.py
Input: uploaded file (Streamlit UploadedFile), source_type (str)
Output: raw_text (str)
chunker.py
Input: raw_text (str), filename (str), source_type (str)
Output: List of dicts: {chunk_id, text, filename, source_type}
embedder.py
Input: chunks (List[dict])
Output: embeddings (np.ndarray shape [n_chunks, 384])
clusterer.py
Input: chunks (List[dict]), embeddings (np.ndarray)
Output: clusters (List[dict]): {cluster_id, representative_chunks[3], all_chunk_ids}
llm.py
Input: clusters (List[dict]), goal (str)
Output: ost_json: {goal, opportunities[{jtbd, solutions[{label, assumptions}], priority_score}]}

Daily standup format (15 min max)
Each person answers 3 questions:
  • What did I complete since yesterday?
  • What will I work on today?
  • Any blocker that needs the PM's decision?

PM closes with: any priority changes, scope decisions, help needed. Run async in Slack if not meeting in person.
Definition of done (per task)
A task is done when:
  • Code is merged to main (via PR, not direct push)
  • It doesn't break any existing feature
  • PM or one other teammate has seen it work
  • The checkbox in this plan is ticked

Partial work = not done. If a task is too big, split it in the daily.
Git workflow
  • One branch per feature: feature/ost-tree, fix/embedding-crash
  • Never push directly to main
  • Small PRs (1 feature = 1 PR). Easier to review.
  • Tag the PM in any PR that changes pipeline output format
  • Use GitHub Issues for bugs: P1 blocker / P2 important / P3 nice-to-have
PM scope decisions (sign-off required)
The PM must approve before the team builds:
  • Any change to what the LLM outputs
  • Any new page or view added to the app
  • Any change to the OST data structure
  • Any new dataset or data source added

For anything else: build it, show it in the daily. Speed over process on small things.
Communication norms
How we communicate as a team:
  • Primary channel: Slack (#discovery-lens-team)
  • Reply within 4 hours during working hours
  • Use threads for technical discussions — keep main channel clean
  • If blocked for >2h, ping the team immediately — don't wait for standup
  • Decisions made in standup must be written down in #decisions within 1h
💡 Tips for working with the LLM
Keep prompts effective and deterministic:
  • Always use temperature=0 for JSON output calls — reduces hallucinations
  • Few-shot examples in the system prompt are non-negotiable for JTBD framing
  • Log every prompt version in prompts/ with a date prefix
  • If the 8B model fails 2× in a row, fall back to 70B — don't fight the model
  • Never parse JSON outside a try/except — Groq output can be subtly malformed
